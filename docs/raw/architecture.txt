## 2. Process Model

[Diagram: Multi-Process Model. Shows a central 'UI Process' (Main Process) managing multiple 'Renderer Processes' (one per tab). A separate 'GPU Process' handles all Metal rendering commands for all tabs, and a 'Network/ML Process' handles shared background tasks. Communication occurs via Mach Ports and shared memory.]

| Process | Responsibilities | Sandboxed |
|---|---|---|
| **UI Process** | Window management, tab bar, URL bar, user input dispatch | Minimal |
| **Renderer (×N)** | HTML parsing, CSS resolution, layout, JS execution, paint | **Yes** — no filesystem/network |
| **GPU Process** | Metal command encoding, texture management, compositing | Yes |
| **ML Process** | Core ML inference, MLX LLM, cache predictions | Yes |

## 3. Memory Architecture

### 3.1 Unified Memory Strategy

[Diagram: Unified Memory Strategy. Illustrates Apple Silicon's UMA where CPU and GPU share the same physical memory pool. The diagram shows how a 'Texture Resource' created by the CPU is immediately accessible by the GPU without copying, drastically reducing latency and memory footprint.]

### 3.2 Arena Allocation Per Tab

[Diagram: Arena Allocation. Shows a 'Memory Pool' dedicated to a single tab. As the tab parses HTML and CSS, 'DOM Nodes' and 'Style Objects' are stacked linearly in the arena. When the tab is closed, the 'Entire Arena' is wiped in a single O(1) operation, ensuring zero leaks.]

### 3.3 Allocator Hierarchy

| Allocator | Scope | Freed When |
|---|---|---|
| **Page Allocator** | Large assets (images, WASM binaries, IOSurface) | Asset evicted |
| **Arena Allocator** | Per-tab DOM, CSS, layout data | Tab closed / navigated |
| **Fixed Buffer** | HTTP headers, IPC messages, small strings | End of function scope |
| **GPA (debug only)** | All allocations during development | Leak detection |

## 4. Rendering Pipeline

[Diagram: Rendering Pipeline. A flow chart: 
1. DOM + CSSOM -> Style Resolution.
2. Resolution -> Layout Tree (Box Model).
3. Layout Tree -> Paint Commands (Vertex/Index data).
4. Paint Commands -> Metal Command Buffer.
5. Metal Command Buffer -> GPU -> Screen.]

### 4.1 Frame Lifecycle (120 Hz target = 8.33 ms budget)

[Diagram: Frame Lifecycle Timeline. A horizontal bar spanning 8.33ms, divided into segments: 
- Input Handling (0.5ms).
- JS Execution (2.0ms).
- Style/Layout (2.0ms).
- Paint Encoding (1.5ms).
- GPU Work (2.0ms).
- Idle/Buffer (0.33ms).]

## 5. JavaScript Engine Architecture

[Diagram: JS Engine Architecture. Shows a dual-engine setup. 'JavaScriptCore' is the primary engine for high-performance active tabs, while 'QuickJS' acts as a low-memory fallback for background tabs. A 'Bridge' layer handles DOM bindings and IPC between the JS environments and the Core Engine.]

## 6. Networking & Resource Loading

[Diagram: Network Stack. Shows the 'Resource Loader' interacting with 'NSURLSession'. It includes a 'Cache' layer (Volatile Arena + Persistent Disk) and an 'ML Predictor' that suggests URLs to prefetch based on history. Data flows from the wire into the HTML Parser in chunks.]

## 8. Component Dependency Matrix

| Component | Depends On | Depended By |
|---|---|---|
| `platform/` | AppKit, Metal, macOS APIs | Everything |
| `dom/` | `platform/` (allocators) | `css/`, `layout/`, `js/`, `devtools/` |
| `css/` | `dom/` | `layout/` |
| `layout/` | `css/`, `dom/` | `render/`, `devtools/` |
| `render/` | `layout/`, `platform/` (Metal) | `ui/`, `ipc/` |
| `js/` | `dom/`, `platform/` (JSC/QuickJS) | `ui/` |
| `net/` | `platform/` (NSURLSession) | `dom/`, `ui/` |
| `ipc/` | `platform/` (Mach, shm) | `render/`, `net/`, `ui/` |
| `ui/` | `render/`, `net/`, `js/` | User-facing |
| `ml/` | `platform/` (CoreML, MLX) | `net/`, `devtools/` |
| `devtools/` | `dom/`, `layout/`, `render/`, `js/` | User-facing |
